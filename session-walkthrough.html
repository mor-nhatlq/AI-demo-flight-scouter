§ 02 — Các prompts

Từng tin nhắn, theo thứ tự đã gửi.

Prompt của user được giữ nguyên văn — kể cả lỗi chính tả. Giá trị nằm ở mức độ ngắn gọn và định hướng của từng tin nhắn, không phải ở độ chau chuốt.

P1
00:00 Mở đầu hội thoại skill · /brainstorm

Tập trung vào nói rõ mục tiêu.

Tôi muốn làm một website nho nhỏ, mục tiêu là list ra các chuyến bay và giá thành dựa vào ngày đi, ngày về, điểm đi, điểm đến và option khứ hồi. Mục tiêu của dự án này là để demonstrate sức mạnh của AI trong việc tự động hoá phần tìm hiểu yêu cầu và coding, và tôi muốn có thể làm một cái live demo coding nho nhỏ dựa vào tài liệu yêu cầu mà tôi với bạn sẽ tạo ngay bây giờ. Cùng tìm hiểu nào!
Ý định
Định hình dự án bằng lý do tồn tại trước khi đả động đến stack hay danh sách tính năng.
Skill
/brainstorm · Skill /brainstorm tự viết. Áp dụng các quy chuẩn brainstorm như YAGNI/KISS/DRY, bắt buộc hỏi người dùng để làm rõ trước khi code.
Kết quả
AI khởi tạo dự án, sau đó hỏi bốn câu hỏi (nguồn dữ liệu, stack, thời lượng demo, khán giả) qua AskUserQuestion.
Phản hồi của AI
"Tuyệt vời! Một dự án demo rất hay — website tìm kiếm chuyến bay để showcase AI-driven development."
Sau đó, AI khởi tạo Explore subagent để tạo project, gọi AskUserQuestion với 4 câu: (1) nguồn dữ liệu — mock / real API / hybrid (2) tech stack — Next.js / React+Vite / vanilla (3) thời lượng demo — 15-20 / 30-45 / 60+ phút (4) khán giả — business / developers / mixed.
Bài học
Mở đầu bằng tại sao, không phải cái gì. Cho AI biết lý do kinh doanh thì nó sẽ đề xuất scope phù hợp, thay vì implement luôn yêu cầu mình gõ ra đầu tiên.
Sau P1 AI scout workspace (qua Explore subagent), xác nhận project trống, rồi trả về 4 câu hỏi trắc nghiệm cần user chốt trước khi đi tiếp: nguồn dữ liệu (mock / real API / hybrid), tech stack (Next.js / React+Vite / vanilla), thời lượng demo (15-20 / 30-45 / 60+ phút), khán giả (business / developers / mixed). Câu trả lời của user nằm trong prompt kế tiếp.
P2
00:02 Trả lời bốn câu hỏi tool · AskUserQuestion

Ràng buộc giá trị hơn tính năng.

Lưu ý về format: User không gõ text tự do ở P2. Trả lời qua UI AskUserQuestion — mỗi câu chọn 1 option (hoặc "Other" để gõ custom). Với câu "Nguồn dữ liệu" và "Thời lượng demo", user chọn "Other" để thêm ràng buộc riêng. Card này hiển thị lại các câu trả lời đã submit.

Nguồn dữ liệu · Other
Gọi API thật, có option API nào khác ko, ví dụ như từ OpenAPI hay Google Flights. Tìm hiểu giúp tôi xem có các free option nào nha.
Tech Stack
React + Vite
Thời lượng demo · Other
30 phút tối đa, có search + results + filter/sort. UI bắt mắt và giữ đơn giản. Có hiển thị thời tiết trong khoảng thời gian mà người dùng chọn tại điểm đến nữa.
Khán giả
Business / Management
Ý định
Khoá scope bằng cách gắn ràng buộc định lượng và tính chất vào từng lựa chọn.
Kết quả
AI lúc này đã có: ưu tiên API thật, stack tối giản, ngân sách thời gian 30 phút, một yêu cầu ẩn (panel thời tiết), và tín hiệu về khán giả (visual impact > chiều sâu kỹ thuật).
Phản hồi của AI
"Tốt lắm! Bạn muốn: Real API cho flight data — tôi sẽ research các free option. React + Vite — gọn nhẹ, phù hợp demo. 30 phút — search + results + filter/sort + weather tại điểm đến. Business audience — focus visual impact + tốc độ delivery. Để tôi research các Flight API miễn phí và Weather API cho bạn." Ngay sau đó dispatch 2 researcher subagent song song ở chế độ nền.
Bài học
Câu trả lời ngắn vẫn hiệu quả nếu kèm ràng buộc. "30 phút tối đa", "bắt mắt và giữ đơn giản", cộng với một yêu cầu bất ngờ (thời tiết) — ba mảnh vụn này truyền tải nhiều thông tin thiết kế hơn ba đoạn văn.
Sau P2 Trong cùng một lượt trả lời, AI làm ba việc: (1) xác nhận ngắn gọn các lựa chọn, (2) đưa ra nhận định — user muốn API thật nhưng "có option nào khác không" là gợi ý cần tìm hiểu lựa chọn free, (3) dispatch hai researcher subagent song song chạy nền để so sánh flight APIs và weather APIs. Không cần prompt riêng để kích hoạt research — AI chủ động làm vì trong P2 user đã yêu cầu "tìm hiểu giúp tôi xem có các free option nào".
P3
00:03 Không có prompt — agent tự khởi động agent · researcher × 2

Research song song, nằm ngoài main context.

Không cần user yêu cầu, main agent dispatch hai subagent researcher chạy nền — một cho free flight-search APIs, một cho free weather APIs. Hai truy vấn hoàn toàn độc lập, nên chạy song song tiết kiệm khoảng 40 giây wall time và giữ các bảng so sánh dài ra khỏi main context window.

Agent
researcher · thu thập thông tin kỹ thuật, tạo báo cáo so sánh ngắn gọn, trích nguồn đầy đủ.
Dispatch
(a) Free flight APIs — Kiwi Tequila, Amadeus, Skyscanner, AviationStack, FlightAPI.io, Google Flights. (b) Free weather APIs — Open-Meteo, Visual Crossing, Tomorrow.io, OpenWeatherMap, WeatherAPI.
Kết quả
Hai báo cáo được lưu trong plans/reports/. Người thắng: Kiwi Tequila cho chuyến bay, Open-Meteo cho thời tiết (không cần API key).
Phản hồi của AI
Sau khi dispatch, AI thông báo "Đang research song song Flight APIs và Weather APIs miễn phí. Sẽ có kết quả ngay." Khi hai agent trả về, AI tổng hợp: "Flight API: Kiwi Tequila API (Winner) — Free, không cần credit card, real-time pricing. Weather API: Open-Meteo (Winner) — Free, không cần API key luôn — zero friction cho demo." Rồi tiếp tục hỏi thêm 3 câu về UI style, filter scope, ngôn ngữ.
Bài học
Khi hai câu hỏi không phụ thuộc vào nhau, dispatch song song. Main thread tiếp tục hội thoại; kết quả về qua notification. Đây là cách tiết kiệm wall time lớn nhất trong live demo.
Sau P3 Khi cả hai researcher trả kết quả, AI tổng hợp thành bảng winners: Kiwi Tequila API cho chuyến bay (free tier không cần credit card, real-time) và Open-Meteo cho thời tiết (không cần API key, forecast 16 ngày). Xong phần tech, còn 3 quyết định thiết kế chưa chốt — AI gọi AskUserQuestion lần hai, lần này kèm ASCII preview mockup đặt cạnh mỗi option: phong cách UI (glassmorphism / clean minimal / dashboard), phạm vi filter, ngôn ngữ UI.
P4
00:05 Quyết định thiết kế tool · AskUserQuestion + preview

Cho AI show ba option trực tiếp trên màn hình.

Phong cách UI (chọn từ ba ASCII mockup)
Modern glassmorphism — background gradient, glass cards, blur effects.
Filter
Chỉ Price sort + Airline filter. (Áp dụng KISS — bỏ cabin class, bỏ duration band.)
Ngôn ngữ app
UI tiếng Anh.
Ý định
Chốt nhanh ba quyết định thiết kế còn lại mà không cần mô tả UI bằng lời.
Công cụ
AskUserQuestion hỗ trợ khối preview đặt cạnh — mỗi option đính kèm một ASCII mockup inline của layout.
Kết quả
Glassmorphism được chọn trong vài giây thay vì cả đoạn mô tả dài, và AI có thể show mockup ngay cạnh nhãn option.
Phản hồi của AI
AI tổng hợp Design Summary thành bảng: Purpose, Stack, Flight API, Weather API, UI Style, Filters, Language, Demo time. Liệt kê 5 features trong scope + kiến trúc SPA. Note quan trọng: "Không cần backend — tất cả gọi API trực tiếp từ client (Kiwi cho phép CORS, Open-Meteo không cần key)." Rồi hỏi "Bạn có muốn thêm/bớt gì không? Nếu OK thì tôi sẽ tạo tài liệu yêu cầu (brainstorm report) và tiếp tục sang plan."
Bài học
Chọn thiết kế nhanh hơn khi có bất kỳ hình ảnh nào — kể cả ASCII. Nếu model có thể render preview ngay trong câu hỏi, dùng nó. Mô tả layout bằng lời thường thiếu chi tiết.
Sau P4 AI tổng hợp toàn bộ quyết định thành một Design Summary: bảng Purpose / Stack / APIs / UI Style / Filters / Language / Demo time, cộng với 5 features trong scope và sơ đồ kiến trúc SPA (không cần backend vì cả hai API đều CORS-friendly). Cuối cùng AI hỏi: "Bạn có muốn thêm/bớt gì không? Nếu OK thì tôi sẽ tạo tài liệu yêu cầu (brainstorm report) và tiếp tục sang plan." Đây là moment quan trọng — user chưa sẵn sàng commit ngay.
P5
00:07 Đổi hướng giữa chừng

Hai tin nhắn liên tiếp: do dự, rồi chốt.

Sau khi AI hỏi "Bạn có muốn thêm/bớt gì không?", user gửi hai tin nhắn cách nhau vài giây. Tin thứ nhất mang tính thăm dò — báo rằng chưa 100% hài lòng nhưng cũng không muốn liệt kê cụ thể phải sửa gì.

Cần chỉnh sửa

AI phản ứng trung tính: "Bạn muốn chỉnh sửa gì? Cứ nói thoải mái — thêm feature, bớt feature, đổi style, hay thay đổi approach gì cũng được!" Không phán đoán, không tự sửa. Mở cửa cho user suy nghĩ thêm. Trong lúc đó, user nhận ra rằng nhìn plan chi tiết sẽ dễ phản biện hơn là tranh luận trong không khí — nên pivot sang yêu cầu xuất file:

À thôi, xuất cho tôi cái plan đi để tôi review. Ném vào folder FlightScouter nha
Ý định
Đổi chiến thuật: thay vì tranh luận trừu tượng, chuyển sang review một artifact cụ thể (plan.md) nơi có thể comment từng dòng.
Kết quả
AI viết hai file markdown vào folder dự án: brainstorm-report.md (problem statement + decisions table + evaluated alternatives + risks + success criteria) và plan.md với YAML frontmatter status: pending, chi tiết 6 phase — mỗi phase có step-by-step, danh sách file cần sửa/tạo, tiêu chí hoàn thành, thời lượng ước tính, dependency giữa các phase.
Phản hồi của AI
Sau khi ghi 2 file, AI trả về outline của 6 phase đã viết (Scaffold → Search Form → Flight API + Results → Filter/Sort → Weather Panel → Glassmorphism Polish), tổng thời lượng ~30 phút. Kết: "Review và cho tôi biết muốn chỉnh gì nhé! Khi OK tôi sẽ bắt đầu implement." Chuyển quyền kiểm soát lại cho user — không implement cho đến khi user cho phép.
Bài học
Khi do dự, đừng ép mình phải biết đang do dự vì điều gì. Một câu "cần chỉnh sửa" rồi pivot sang "xuất plan ra file để tôi review" tốn ít năng lượng hơn là cố gắng diễn đạt chính xác đang lăn tăn chỗ nào — và thường cho kết quả tốt hơn vì bạn review được artifact cụ thể thay vì không khí.
Sau P5 User mở plan.md đọc qua. Plan trông ổn nhưng user muốn biết AI định thực thi nó thế nào — plan chỉ kể ra cái gì cần làm, chưa nói rõ cách sẽ làm (sequential hay parallel? có kiểm tra giữa chừng không? dùng subagent nào?). Thay vì đọc mò, user hỏi thẳng.
P6
00:08 Audit kế hoạch

Bắt AI mô tả workflow của chính nó.

Bạn nói cho tôi workflow implement của bạn sẽ thế nào?
Ý định
Bắt lỗi trong kế hoạch khi sửa còn rẻ.
Kết quả
AI giải thích: 6 phase tuần tự (mỗi phase là một dispatch fullstack-developer), lý do tuần tự chứ không song song (mọi phase đều sửa App.tsx), build check giữa phase, code review cuối cùng.
Phản hồi của AI
AI trả lời bằng diagram ASCII flow: Phase 1 Scaffold (3 phút) → Phase 2 Search Form (5 phút) → Phase 3 Flight API + Results (8 phút) → Phase 4 Filter/Sort (4 phút) → Phase 5 Weather Panel (5 phút) → Phase 6 Glassmorphism Polish (5 phút). Giải thích rõ "Tại sao tuần tự? Mỗi phase phụ thuộc phase trước. Parallel sẽ gây conflict vì cùng edit App.tsx. Tuần tự = mỗi step build on top of previous = ít bug hơn." Và kế hoạch verify sau mỗi phase + code review cuối.
Bài học
Một đoạn "bạn sẽ làm gì tiếp theo" của AI sẽ phơi bày các hiểu lầm mà nếu không sẽ biến thành merge conflict một tiếng sau.
Sau P6 AI mô tả workflow bằng diagram ASCII + giải thích sequencing. User hài lòng với chiến lược nhưng chưa biết AI sẽ kích hoạt công cụ cụ thể nào — một lớp chi tiết nữa cần audit trước khi bấm "Chạy". Câu hỏi tiếp theo xuống một tầng: từ workflow sang instrumentation.
P7
00:09 Audit công cụ

Biết trước agent và skill nào sẽ chạy.

Bạn sẽ chạy các con agent gì và sử dụng skill gì?
Ý định
Xem trước danh sách công cụ. Cái nào bất ngờ là cơ hội để veto.
Kết quả
AI liệt kê fullstack-developer, code-reviewer, code-simplifier, cùng các skill cook, fix, simplify, docs-seeker, react-best-practices, kèm điều kiện kích hoạt từng cái.
Phản hồi của AI
AI trả lời bằng 3 bảng: (1) Agents — fullstack-developer (implement 6 phase), code-reviewer (review sau Phase 6), code-simplifier (nếu reviewer phát hiện code phức tạp). (2) Skills — cook (checklist mỗi phase), fix (bugs), simplify (post-phase review), docs-seeker (tra Kiwi/Open-Meteo docs), react-best-practices. (3) Verification trực tiếp — npm run build, npm run dev, npx tsc --noEmit sau mỗi phase. Kết: "Bắt đầu implement không?"
Bài học
Minh bạch tạo khả năng kiểm soát. Danh sách agent và skill AI định dùng là bản nháp thứ hai của kế hoạch — review nó giống như bạn review đoạn văn.
Sau P7 AI liệt kê 3 agents (fullstack-developer, code-reviewer, code-simplifier) + 5 skills và điều kiện kích hoạt. Mọi thứ đã rõ: plan có, workflow có, danh sách công cụ có. Không còn gì để audit. Lúc này một câu lệnh ngắn là đủ.
P8
00:10 Thực thi skill · /cook

Hai chữ là đủ nếu context đã rõ.

Chạy đi
Skill
/cook · smart feature implementation — chọn code-mode vì plan.md đã có, nên bỏ qua bước research.
Công việc
Phase 1 — Scaffold Vite (làm trong thread).
Phase 2 — Component SearchForm (fullstack-developer).
Phase 3 — Flight API service + Results (fullstack-developer).
Phase 4 — Filter/Sort (làm trong thread, scope nhỏ).
Phase 5 — Weather panel (fullstack-developer).
Phase 6 — Glassmorphism polish + font Inter + animation (trong thread).
Verify
npx tsc --noEmit + npm run build giữa mỗi phase. Bắt được một lỗi type-import do verbatimModuleSyntax và fix trước khi đi tiếp.
Kết quả
SPA chạy: build sạch, theme glassmorphism, skeleton loader, animation fade-in cho danh sách, pipeline city → weather hoạt động.
Phản hồi của AI
"Let's go! Bắt đầu Phase 1 — Scaffold project." Invoke /cook, tạo TodoWrite 7 items. Phase 1 gặp lỗi "directory not empty" vì có sẵn brainstorm-report.md + plan.md — AI tự fix bằng cách scaffold Vite vào /tmp rồi copy vào. Các phase kế tiếp mỗi cái là một fullstack-developer dispatch với acceptance criteria rõ ràng. Giữa Phase 2 và 3, AI bắt lỗi verbatimModuleSyntax type-import, fix ngay, verify build lại. Sau Phase 6, dispatch code-reviewer tìm ra 3 issues: .env chưa gitignore, thiếu null guard trong weather mapping, duplicate SearchParams type. Apply 2 fix đầu (fix #3 bỏ qua vì là demo).
Bài học
Khi đã đầu tư plan và approval rõ ràng từ trước, prompt thực thi có thể gần như trống. "Chạy đi" hoạt động được vì plan.md, danh sách agent, và tiêu chí hoàn thành đã được chốt trước đó.
Sau P8 ~15 phút implementation trôi qua — 6 phase xong, code review xong, 3 fix đã apply, build sạch. AI báo "Done! Cách chạy: npm run dev." User nhìn code, nhận ra flight-api.ts đang đọc VITE_KIWI_API_KEY từ .env — nhưng trong suốt session chưa ai nói đến việc đăng ký lấy key. Đây là phần AI không thể tự làm hộ.

00:30 Câu hỏi ngoài lề

Hỏi về những phần AI không làm.

Tôi lấy API key của Kiwi ở đâu vậy?
Ý định

§ 03 — Skills

Danh sách công cụ, đặt tên rõ ràng.

Skill là những hợp đồng markdown nhỏ thay đổi cách main agent hành xử. Bản thân skill không spawn subagent; chúng định hướng main thread.

/brainstorm skill
Brainstorm giải pháp theo YAGNI/KISS/DRY. Bắt buộc có design-before-code gate và dùng AskUserQuestion để làm rõ scope.
/cook skill
Smart feature implementation. Tự phát hiện code-mode khi có plan file, ngược lại chạy research → plan → implement → test → review.
/frontend-design skill
Tạo web interface production-grade. Kèm quy tắc anti-slop (tránh Inter, tránh AI-purple gradient) và nghi thức design-thinking.
AskUserQuestion tool
Input multiple-choice có cấu trúc, kèm preview inline. Tối đa bốn câu mỗi lần gọi; một option "Other" mở sẵn.
TodoWrite tool
Task list phạm vi session. Luôn đồng bộ để hiển thị tiến độ và báo phase nào đang chạy.
react-best-practices skill
Hướng dẫn performance + pattern cho React/Next.js. Tham khảo khi viết component Suspense-aware và effect cleanup.
fix skill
Fix bug theo root-cause-first. Giữ sẵn cho lỗi compile/runtime giữa các phase.
simplify skill
Review các thay đổi gần đây để tái sử dụng và dễ đọc. Giữ sẵn như lớp post-implementation; phiên này chưa cần tới.
docs-seeker skill
Truy vấn context7 / llms.txt để lấy doc library mới nhất. Hữu ích cho Tailwind v4 quirks và endpoint Open-Meteo.
preview skill
Sinh visual explanation và HTML artifact. Surface thay thế cho frontend-design khi cần diagram nhanh.
